## What changed in the landscape

This baseline snapshot does not compare against an older in-repo snapshot. It records the currently visible pattern center of gravity.

The clearest signals are:

- skill guidance is converging on concise spines plus on-demand supporting files
- prompt and skill guidance are both leaning harder on explicit workflows, examples, and evaluation
- portability is increasingly achievable through the Agent Skills format, but platform-specific extensions still exist and need isolation

## Candidate patterns

### Pattern: concise spine plus progressive disclosure

- observed signal: Strong repeated emphasis on keeping the always-loaded instructions compact and moving detailed material into supporting files.
- source support: Anthropic skill-authoring guidance; Claude Code skills docs.
- likely benefit: Better context efficiency, clearer maintenance boundary, easier cross-agent portability.
- likely risk: Important details may become too hidden if the split is careless.
- proposed status: `adopt`
- CraftKit implication: Keep `SKILL.md` under a disciplined size budget and plan `references/` early when depth is real.

### Pattern: calibrated degree of freedom

- observed signal: Recent skill guidance is explicit that instruction specificity should match task fragility.
- source support: Anthropic skill-authoring guidance.
- likely benefit: Fewer brittle workflows for open-ended tasks and fewer accidents in fragile workflows.
- likely risk: Over-application can create needless exactness or, in the other direction, vague skills.
- proposed status: `adopt`
- CraftKit implication: Make freedom-level choice a first-class design decision in skill scaffolding.

### Pattern: explicit workflow and checklisting

- observed signal: Multi-step tasks are increasingly specified as clear sequences, sometimes with checklist progress tracking.
- source support: Anthropic skill-authoring guidance; OpenAI instruction-writing guidance.
- likely benefit: Better execution reliability for complex tasks.
- likely risk: Can become ceremonial if used for simple tasks.
- proposed status: `adopt`
- CraftKit implication: Use steps by default for genuinely multi-phase skills; reserve checklists for the complex end.

### Pattern: trigger-oriented descriptions

- observed signal: Discovery quality depends on naming the task and its trigger contexts directly in metadata.
- source support: Anthropic skill-authoring guidance; OpenAI GPT configuration guidance.
- likely benefit: Better automatic invocation and easier reuse.
- likely risk: Description bloat if every alias is shoved in without discipline.
- proposed status: `adopt`
- CraftKit implication: Continue treating frontmatter `description` as the trigger surface, not an afterthought.

### Pattern: examples for brittle boundaries

- observed signal: Examples are repeatedly recommended when the acceptable output boundary is easy to misread.
- source support: OpenAI instruction-writing guidance; Anthropic examples pattern and related prompting guidance.
- likely benefit: Reduces ambiguity without needing long explanations.
- likely risk: Low-value examples can bloat the spine and overfit wording.
- proposed status: `adopt`
- CraftKit implication: Prefer one compact example over long explanatory prose when behavior is easier to show than define.

### Pattern: eval-first tightening

- observed signal: Prompt and skill guidance is aligning around defining success criteria, testing them, and tightening instructions before adding more tooling.
- source support: Anthropic prompt-engineering overview; Anthropic skill-authoring guidance; OpenAI GPT testing guidance.
- likely benefit: More stable improvement loop, less premature complexity.
- likely risk: Can feel heavier than needed for tiny one-off assets.
- proposed status: `adopt`
- CraftKit implication: Treat eval thinking as default for reusable skills, not only for late optimization.

### Pattern: too many parallel options in one skill

- observed signal: Recent guidance warns against presenting too many options without a clear default.
- source support: Anthropic skill-authoring guidance.
- likely benefit of rejecting it: Keeps the skill executable instead of encyclopedic.
- likely risk if copied: A skill becomes a menu instead of a workflow.
- proposed status: `avoid`
- CraftKit implication: Favor one primary path and isolate alternatives behind explicit conditions.

### Pattern: time-sensitive operational advice in the core spine

- observed signal: Current guidance warns against embedding fast-staling information into durable instructions.
- source support: Anthropic skill-authoring guidance.
- likely benefit of rejecting it: Lower maintenance burden and better long-term correctness.
- likely risk if copied: Core skills decay quickly and lose trust.
- proposed status: `avoid`
- CraftKit implication: Keep volatile guidance in dated radar snapshots or isolated guides, not in the main skill spine.

### Pattern: platform-specific invocation extensions

- observed signal: The open format is portable, but product-specific invocation controls and extensions are growing around it.
- source support: Claude Code skills docs; Agent Skills overview.
- likely benefit: Better ergonomics in a specific platform.
- likely risk: Lock-in if treated as universal.
- proposed status: `watch`
- CraftKit implication: Track these features, but keep CraftKit's core design portable unless a skill is intentionally platform-bound.

### Pattern: heavier executable-code packaging

- observed signal: Strong guidance now exists for utility scripts and executable support, but it assumes more about runtime and packaging.
- source support: Anthropic skill-authoring guidance.
- likely benefit: Deterministic execution for repeated fragile tasks.
- likely risk: More maintenance, more environment coupling, weaker portability.
- proposed status: `watch`
- CraftKit implication: Use scripts selectively; do not let "can package code" become "should package code."

## Adopt this cycle

- concise spine plus progressive disclosure — strong, repeated signal with direct benefit to CraftKit's context budget
- calibrated degree of freedom — clarifies a design choice CraftKit already makes informally
- explicit workflow and checklisting — useful default for complex skills, not for everything
- trigger-oriented descriptions — directly improves discoverability and reuse
- examples for brittle boundaries — compact examples often outperform longer explanations
- eval-first tightening — aligns with CraftKit's existing autoresearch orientation

## Avoid this cycle

- too many parallel options in one skill — turns a skill into a menu instead of an executable guide
- time-sensitive operational advice in the core spine — decays too quickly for durable assets
- monolithic all-in-one skill files — implicit corollary of the progressive disclosure signal
- provider-specific wording in the portable core — conflicts with CraftKit's cross-agent goal

## Watch, not default

- platform-specific invocation extensions — likely useful, but still too tied to host-product behavior
- heavier executable-code packaging — beneficial for some tasks, but easy to overuse
- visual and intermediate-output validation patterns — directionally promising, still uneven across setups
